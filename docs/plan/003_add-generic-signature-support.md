# Add Generic Signature Support to the Scanner

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [`.agent/PLANS.md`](/home/toliner/projects/spector/.agent/PLANS.md).

## Purpose / Big Picture

After this change, Spector will return type information that preserves generic type parameters and bounds instead of collapsing them to raw classes. Today the scanner can identify classes, methods, fields, inheritance, and some Kotlin metadata, but the generic-signature tests in `src/test/kotlin/dev/toliner/spector/scanner/ClassScannerTest.kt` explicitly note that type parameter extraction is not implemented. For an AI agent or a developer trying to understand library APIs, `List<String>` and `List<T>` are materially different from raw `List`, so this is a core data-quality improvement.

## Surprises & Discoveries

- Observation: The scanner already captures the raw `signature` string in `ClassScanner.ClassInfoVisitor`, but it never uses it.
  Evidence: `ClassScanner.kt` stores `signature` during `visit(...)` and ignores it in `buildClassScanResult()`.
- Observation: The test suite already points at this missing behavior.
  Evidence: `ClassScannerTest.kt` contains TODO comments for `Map` and `Class<T>` type parameter extraction.
- Observation: The current `TypeRef` model already represents the generic structures needed for this milestone, including type variables, wildcard variance, bounds, and arrays.
  Evidence: `TypeRef.kt` exposes `TYPEVAR`, `WILDCARD`, `bounds`, and `args`, so no schema expansion was required.
- Observation: Running two `./gradlew test --tests ...` commands in parallel can collide on Gradle's XML test-result files even when they target different specs.
  Evidence: parallel execution failed with `Could not write XML test results for dev.toliner.spector.storage.TypeIndexerTest`, while the same command succeeded immediately when rerun alone.

## Decision Log

- Decision: Prioritize generic support after workflow stabilization and Gradle indexing.
  Rationale: This is a core correctness improvement, but the system is easier to adopt once users can reliably run it and index external projects. After that, improving the fidelity of indexed data has the highest long-term value.
  Date/Author: 2026-03-17 / Codex
- Decision: Preserve existing raw-descriptor behavior as a fallback when signature data is absent.
  Rationale: Some class files omit generic signatures. The scanner must remain robust across older or generated bytecode.
  Date/Author: 2026-03-17 / Codex
- Decision: Keep `ClassInfo.superClass` and `ClassInfo.interfaces` as raw FQCN strings for this milestone instead of changing the public model to carry generic supertypes.
  Rationale: The existing API, storage schema, and hierarchy queries rely on string-based inheritance links. This milestone focuses on preserving generic information where the model already supports it: type parameters and member type references.
  Date/Author: 2026-03-17 / Codex
- Decision: Add a Java test fixture under `src/test/java` rather than relying only on JDK classes for member-level generic assertions.
  Rationale: JDK classes prove real-world signatures such as `Map<K, V>` and `Class<T>`, but a local fixture gives a stable, readable case for wildcard field types, bounded method type parameters, and generic throws.
  Date/Author: 2026-03-17 / Codex

## Outcomes & Retrospective

The scanner now interprets JVM `Signature` attributes for classes, fields, and methods and stores the richer result in the existing model. `Map<K, V>` and `Class<T>` now expose non-empty `typeParameters`, the Java fixture proves wildcard and method-generic preservation, and SQLite round-trips the resulting structures without extra migration work.

What remains out of scope is generic information on superclass and interface links. `ClassInfo.superClass` and `ClassInfo.interfaces` still expose raw class names, so a type such as `AbstractList<E>` is still represented as `"java.util.AbstractList"` in hierarchy APIs. That tradeoff kept this change compatible with the current storage and query surface while closing the biggest scanner fidelity gap.

## Context and Orientation

The class-file scanner lives in `src/main/kotlin/dev/toliner/spector/scanner/ClassScanner.kt`. The normalized model types live under `src/main/kotlin/dev/toliner/spector/model/`, especially `TypeRef.kt`, `ClassInfo.kt`, and `MemberInfo.kt`. The SQLite persistence layer serializes those model objects directly in `src/main/kotlin/dev/toliner/spector/storage/TypeIndexer.kt`, so once the model contains richer generic information, it will be stored automatically.

In JVM bytecode, the "descriptor" is the compact type string that the runtime needs, such as `Ljava/util/List;` or `(Ljava/lang/Object;)V`. The optional "Signature" attribute stores richer source-level generic information, such as `Ljava/util/List<Ljava/lang/String;>;`. ASM can parse both. In this repository, descriptor parsing already exists, but signature parsing does not.

## Plan of Work

Begin by confirming the model can represent the target information. If `TypeRef` already supports type variables, arrays, wildcards, and class types with arguments, reuse it; if not, extend it first in a backward-compatible way. Then add a small signature parser, likely using ASM's signature visitor utilities, that converts class, field, and method signatures into the repository's `TypeRef` and type-parameter structures.

Update `ClassScanner` in three places. For classes, parse the class-level `signature` string captured in `visit(...)` to populate `ClassInfo.typeParameters`. For fields, inspect the `signature` argument of `visitField(...)` and prefer it over the raw descriptor when present. For methods, inspect the `signature` argument of `visitMethod(...)` to populate generic parameters, generic return types, thrown type variables, and method-level type parameters when available. Maintain the current descriptor-based fallback path for classes without signature metadata or with malformed signatures.

Once the parser is in place, expand `ClassScannerTest.kt` and, if needed, add fixture classes under `src/test/kotlin` that exercise bounded type parameters, nested generics, wildcards, and generic methods. Add at least one storage or API-level test that proves this richer model survives round-tripping through SQLite and JSON serialization.

## Validation and Acceptance

Acceptance is not "the code compiles"; it is that generic information becomes observable. The implemented tests now assert non-empty type parameter lists for `java.util.Map` and `java.lang.Class`, plus a fixture-backed method-level case with bounded method type parameters, wildcard parameter types, generic return types, and checked exceptions. Existing non-generic scanner tests continue to pass, and the SQLite test proves the richer structures survive serialization and deserialization without changing the JSON schema.

One concrete example from the new fixture is the field type `Map<String, List<? extends Number>>`, which now appears in the scanner model as:

```text
TypeRef(
  kind=CLASS,
  fqcn=java.util.Map,
  args=[
    TypeRef(kind=CLASS, fqcn=java.lang.String),
    TypeRef(
      kind=CLASS,
      fqcn=java.util.List,
      args=[TypeRef(kind=WILDCARD, wildcard=OUT, bounds=[TypeRef(kind=CLASS, fqcn=java.lang.Number)])]
    )
  ]
)
```

## Interfaces and Dependencies

Use ASM's existing signature parsing support rather than writing a custom parser from scratch. The implementation now does this in `src/main/kotlin/dev/toliner/spector/scanner/GenericSignatureParser.kt` by using `SignatureReader` and `SignatureVisitor` to build `TypeRef` and `TypeParameter` values. The public repository-facing interfaces that must remain coherent are `TypeRef`, `ClassInfo`, `FieldInfo`, `MethodInfo`, and the `ClassScanner.scanClass` result. At completion, these types represent and persist generic signatures for class and member metadata without breaking descriptor-only class files.
