# Add Generic Signature Support to the Scanner

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

After this change, Spector will return type information that preserves generic type parameters and bounds instead of collapsing them to raw classes. Today the scanner can identify classes, methods, fields, inheritance, and some Kotlin metadata, but the generic-signature tests in `src/test/kotlin/dev/toliner/spector/scanner/ClassScannerTest.kt` explicitly note that type parameter extraction is not implemented. For an AI agent or a developer trying to understand library APIs, `List<String>` and `List<T>` are materially different from raw `List`, so this is a core data-quality improvement.

- Observation: The scanner already captures the raw `signature` string in `ClassScanner.ClassInfoVisitor`, but it never uses it.
  Evidence: `ClassScanner.kt` stores `signature` during `visit(...)` and ignores it in `buildClassScanResult()`.
- Observation: The test suite already points at this missing behavior.
  Evidence: `ClassScannerTest.kt` contains TODO comments for `Map` and `Class<T>` type parameter extraction.

## Decision Log

- Decision: Prioritize generic support after workflow stabilization and Gradle indexing.
  Rationale: This is a core correctness improvement, but the system is easier to adopt once users can reliably run it and index external projects. After that, improving the fidelity of indexed data has the highest long-term value.

- Decision: Preserve existing raw-descriptor behavior as a fallback when signature data is absent.
  Rationale: Some class files omit generic signatures. The scanner must remain robust across older or generated bytecode.

## Outcomes & Retrospective

This draft focuses on data quality. Success means that queries for well-known generic JDK and Kotlin types return explicit type parameters and bounds in the stored JSON model, and that old non-generic cases continue to work. The result should make the database meaningfully more useful to downstream AI tooling.

## Context and Orientation

The class-file scanner lives in `src/main/kotlin/dev/toliner/spector/scanner/ClassScanner.kt`. The normalized model types live under `src/main/kotlin/dev/toliner/spector/model/`, especially `TypeRef.kt`, `ClassInfo.kt`, and `MemberInfo.kt`. The SQLite persistence layer serializes those model objects directly in `src/main/kotlin/dev/toliner/spector/storage/TypeIndexer.kt`, so once the model contains richer generic information, it will be stored automatically.

In JVM bytecode, the "descriptor" is the compact type string that the runtime needs, such as `Ljava/util/List;` or `(Ljava/lang/Object;)V`. The optional "Signature" attribute stores richer source-level generic information, such as `Ljava/util/List<Ljava/lang/String;>;`. ASM can parse both. In this repository, descriptor parsing already exists, but signature parsing does not.

## Plan of Work

Begin by confirming the model can represent the target information. If `TypeRef` already supports type variables, arrays, wildcards, and class types with arguments, reuse it; if not, extend it first in a backward-compatible way. Then add a small signature parser, likely using ASM's signature visitor utilities, that converts class, field, and method signatures into the repository's `TypeRef` and type-parameter structures.

Update `ClassScanner` in three places. For classes, parse the class-level `signature` string captured in `visit(...)` to populate `ClassInfo.typeParameters`, generic superclasses, and generic interfaces if the model supports them. For fields, inspect the `signature` argument of `visitField(...)` and prefer it over the raw descriptor when present. For methods, inspect the `signature` argument of `visitMethod(...)` to populate generic parameters, generic return types, thrown type variables, and method-level type parameters when available. Maintain the current descriptor-based fallback path for classes without signature metadata.

Once the parser is in place, expand `ClassScannerTest.kt` and, if needed, add fixture classes under `src/test/kotlin` that exercise bounded type parameters, nested generics, wildcards, and generic methods. Add at least one storage or API-level test that proves this richer model survives round-tripping through SQLite and JSON serialization.

Acceptance is not "the code compiles"; it is that generic information becomes observable. At minimum, the tests for `java.util.Map` and `java.lang.Class` must assert non-empty type parameter lists, and at least one method-level generic scenario must be asserted. Existing non-generic scanner tests must continue to pass. If the API or serialized JSON shape changes, add an acceptance example showing a concrete generic type in the response body.

Use ASM's existing signature parsing support rather than writing a custom parser from scratch. At completion, these types must be able to represent and persist generic signatures without breaking descriptor-only class files.
